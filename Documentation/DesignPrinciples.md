# 锻炼YA Design Principles

Status: Active
Last updated: 2026-07-14

## 1. 文档目的

本文档定义锻炼YA的项目级页面设计规范。它将 `frontend-design` 中“基于具体产品做有意图的设计”转化为锻炼YA的实际约束，用于 Duck Home、身体部位选择、Care Routine、动作视频、训练记录和鸭子成长反馈等所有用户界面。

产品范围以 [ProductVision.md](ProductVision.md) 为准，Duck Home 单入口以 [DecisionLog.md](DecisionLog.md) 中 `DL-001` 为准，具体视觉参考的使用边界见 [Design/UIReference.md](Design/UIReference.md)。

---

## 2. 产品视觉核心

### 2.1 设计服务的人

锻炼YA主要服务希望更轻松地照顾身体的用户，尤其包括对传统健身产品感到压力、不熟悉运动术语或需要更温和引导的父母年龄段用户。

界面不应默认用户具备健身知识，也不应通过数字、目标或警告让用户感到被评价。

### 2.2 希望带来的感受

用户打开锻炼YA时，应先感到：

- 这里有一个熟悉的小伙伴在等我
- 我可以从当下能做到的姿势开始
- 今天只做一点点也有意义
- 这是日常身体照顾，不是竞技、考试或医疗判断

视觉语言应温暖、轻松、有生命感、低压力且年龄友好。“温暖”不等于直接套用米色背景、渐变和大圆角；每个色彩、字体、形状和层级都必须从鸭子陪伴与身体照顾的任务中得到解释。

### 2.3 视觉层级

页面的第一层是当下任务与 Duck Mascot 的状态，第二层是帮助用户继续的主操作，第三层才是辅助信息。

信息密度应与用户当前决策相匹配。一个页面不应同时要求用户理解姿态、身体部位、训练目标、历史记录和成长数据。

## 3. Duck Mascot 的 Signature Element 定位

### 3.1 不是装饰，而是产品语言

Duck Mascot 是锻炼YA的 Signature Element，也是用户识别产品、理解当前状态和获得陪伴反馈的主要媒介。

它必须至少承担下列一项职责：

- 表达当前 Standing / Sitting / Lying 姿态
- 引导用户开始或继续一次身体照顾
- 反馈动作完成、坚持或成长
- 在等待、空状态或中断时给出温和方向

如果鸭子在某个页面中没有承担任何交互、状态或情感职责，就不应为了“品牌感”强行放入一只鸭子。

### 3.2 一个页面只保留一个主要视觉重心

鸭子出现时，角色本身应是页面最具辨识度的表现。周围的色彩、背景、卡片、装饰和动效应保持克制，不与 Duck Mascot 争夺注意力。

页面不应同时出现多个同等强势的“主角”，例如大型数据圆环、高对比渐变按钮、多张高饱和卡片和大尺寸鸭子同时出现。

### 3.3 角色表现必须与产品状态一致

- 姿态、表情、道具和环境必须表达真实业务状态。
- 同一状态在不同页面中应使用一致的角色语汇。
- 鸭子不应模仿教练对用户命令、评分或施压。
- 鸭子的成长反馈应表达“我们又照顾了身体一次”，而不是将用户变成被量化的竞技对象。

## 4. 页面设计流程

任何新页面或重大界面改版都必须先完成下列流程，再进入代码实现。

### Step 1: 写出三个前置答案

#### 用户是谁？

不能只写“锻炼YA用户”。必须描述这个页面中的具体用户状态，例如：

> 一位对运动术语不熟悉、今天只想坐着活动肩颈的用户。

#### 页面的唯一主任务是什么？

一个页面只能有一个主任务。例如 Duck Home 的主任务是选择当下适合的照顾姿态，不是同时展示所有训练数据。

#### 为什么每个元素存在？

为主要鸭子、标题、提示、按钮、卡片、图标、数据和动效分别写出职责。如果某个元素不能帮助用户理解状态、做出决定、完成任务或感受陪伴，应删除它。

### Step 2: 建立页面方向

在设计工具或代码之前，先产出一份简短方向：

- **Color**：4–6 个命名色彩及其作用，色彩来自当前照顾场景和鸭子角色，不是任意流行渐变。
- **Type**：明确标题、正文和辅助信息的字体角色与层级，优先清晰、年龄友好，不为装饰牺牲可读性。
- **Layout**：用一句话和简单线框说明信息顺序，先回答用户下一步做什么。
- **Signature**：明确 Duck Mascot 在这个页面中最值得被记住的一个姿态、反馈或交互瞬间。

### Step 3: 去模板化审查

在实现前询问：

- 如果拿掉鸭子和产品名，这个页面是否会被认为任意一个健身 App？
- 是否使用了与用户任务无关的大数字、圆环、渐变、卡片阵列或标签？
- Duck Mascot 是否真正表达了状态，还是只是贴在通用界面上的插图？
- 视觉上最强的元素是否就是当前最重要的产品信息？
- 是否有元素只是为了“显得丰富”而存在？

只要任何一项无法说明，就应返回 Step 2 修正方向。

### Step 4: 实现与两次自评

1. 按已确认的色彩、字体、布局和 Signature Element 实现，不在代码阶段随意变更方向。
2. 以常规字号、较大字号、减少动态效果和主要设备尺寸检查可用性。
3. 截图审查整体层级，删除一个不必要的装饰或竞争注意力的元素。
4. 以真实用户任务走完路径，确认按钮名称、状态反馈和下一步一致。

## 5. 动效原则

### 5.1 动效必须增强情感连接或状态理解

锻炼YA中的动效只有三种合理职责：

- 表达鸭子姿态或页面状态变化
- 确认用户完成了一次有意义的身体照顾
- 通过角色的呼吸、视线、小动作或回应建立陪伴感

如果动效只是让卡片飞入、数字跳动或界面显得“更高级”，应删除。

### 5.2 聚焦一个有意义的瞬间

一个页面优先有一个经过设计的主动效，而不是让所有元素都移动。例如：

- 姿态切换时，鸭子完成可理解的站、坐、躺变化
- 训练完成时，鸭子给出一次温和、明确的回应
- 连续照顾后，角色通过细小成长变化而不是爆炸式奖励建立长期关系

### 5.3 克制、可中断、可降级

- 动效不得阻断主操作或延迟用户进入下一步。
- 手势反馈要跟手，不使用过度弹性或眩晕位移。
- 支持系统“减少动态效果”，减少后仍能通过静态姿态、文字和层级理解状态。
- 装饰性循环动画必须非常轻微，并不得持续争夺注意力。
- 动效不能表达医疗治愈、身体修复保证或过度拟人化承诺。

## 6. 文案原则

### 6.1 从用户认识的世界出发

界面优先使用“站着、坐着、躺着、肩颈、腰背、开始、暂停、完成”等用户能直接理解的词。避免先要求用户理解训练模型、系统字段、专业缩写或健身黑话。

### 6.2 用主动、明确的语言表达操作

- 按钮直接说明点击后会发生什么，例如“开始坐姿照顾”，不使用“提交”或“继续体验”。
- 同一动作在入口、执行中和结果反馈中使用同一名称。
- 一段文案只完成一个职责：标题说明当前任务，说明补充必要信息，按钮表达动作。

### 6.3 陪伴而不评判

推荐语气：

- “今天想用什么姿势照顾身体？”
- “完成了，今天也照顾了自己一点点。”
- “如果现在不方便，可以稍后再来。”

禁止语气：

- “今日目标尚未达成。”
- “再不训练就要中断连续记录了。”
- “你的身体年龄落后了。”
- “完成这套动作即可治愈腰痛。”

### 6.4 错误与空状态要给出方向

错误信息应说明发生了什么以及用户可以怎么做，不使用“出错了”这类模糊文案。空状态是一个温和的下一步入口，不是对用户没有记录的评价。

## 7. 禁止出现的模板化健身 App 设计模式

### 7.1 数据代替产品主角

禁止把大号卡路里、步数、训练时长、连续天数、完成百分比或圆环进度设为首页主视觉。

数据只有在帮助用户做出当前决定时才显示，并且不得压过 Duck Mascot 和当前照顾任务。

### 7.2 通用健身仪表盘

禁止：

- 首页堆叠“今日数据 + 快捷训练 + 课程推荐 + 排行”卡片
- 用底部 Tab 把统计、训练、动作库和成长并列为一级入口
- 以身体部位或训练目标代替 Duck Home 作为首个产品决策
- 为了填满页面而加入无关推荐、活动和信息流

### 7.3 竞技和焦虑机制

禁止：

- 用红色警告、倒计时或损失提示逼迫用户训练
- 把连续打卡中断表达为失败
- 默认展示排名、打败人数、身材对比或身体评分
- 使用“燃脂、虐腹、暴汗、极限”等与产品语气冲突的刺激文案
- 通过羞耻、恐惧或医疗焦虑提高使用率

### 7.4 无理由的流行视觉套件

禁止在没有产品理由时直接使用：

- 高饱和荧光渐变与黑色竞技背景
- 大量半透明玻璃卡片和悬浮光斑
- 通用米色背景、陶土色点缀和高对比衬线标题的“温暖生活方式”套件
- 新闻纸式细线分栏与高密度编号装饰
- 没有真实顺序意义的 `01 / 02 / 03` 编号
- 所有元素都使用大圆角、阴影和弹跳动画

这些形式不是永久禁用的单个视觉属性，但必须能说明它们为什么特别适合当前鸭子场景和用户任务。“现在流行”、“更像高端 App”或“让页面不空”不是有效理由。

### 7.5 把鸭子当作通用贴纸

禁止：

- 在通用健身模板上只增加一张鸭子插图
- 同一页面堆叠多只无状态意义的鸭子
- 让鸭子遮挡主操作、文字或动作指导
- 角色姿态与当前课程姿态或业务状态不一致
- 为了可爱而使用幼稚、含混或不适合父母年龄段用户的交互

## 8. 设计确认清单

设计进入实现前，必须全部回答“是”：

- [ ] 已写明这个页面的具体用户及当下状态。
- [ ] 已写明页面的唯一主任务。
- [ ] 每个可见元素都有明确的产品职责。
- [ ] Duck Mascot 承担了交互、状态或情感职责，而不是装饰。
- [ ] 页面的视觉重心与用户当前任务一致。
- [ ] 去掉产品名后，页面仍能从鸭子状态和交互中被识别为锻炼YA。
- [ ] 动效帮助理解状态或建立情感连接，并支持减少动态效果。
- [ ] 文案使用用户能理解的词，明确说明下一步，不评判用户。
- [ ] 界面没有使用数据、打卡、排名、红色警告或医疗焦虑制造压力。
- [ ] 已删除至少一个无法支持主任务的装饰或次要元素。
